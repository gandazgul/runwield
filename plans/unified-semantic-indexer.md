# High-Level Architecture

Harns will absorb the Mnemosyne memory pipeline and introduce codebase indexing into a single semantic engine powered by
LanceDB and local ONNX models (`@huggingface/transformers`).

There are two separate domains managed by this engine:

1. **The Codebase Index**: Ephemeral, high-turnover chunks generated by Tree-sitter. Stored in `.hns/index/code_chunks`.
2. **The Memory Bank (Mnemosyne V2)**: Persistent, user/agent-generated facts, ADRs, and preferences. Supports both
   `local` (`.hns/index/project_memories`) and `global` (`~/.config/harns/index/global_memories`) scopes.

---

# Sub-System Design

## 1. Tree-Sitter Structural Chunking

Standard semantic search fails on code because naive character chunking slices functions in half.
`src/extensions/indexer/chunker.js` will use `web-tree-sitter` (WASM) to parse the AST.

- **Target Nodes**: The chunker will recursively walk the AST and extract complete nodes of type: `class_declaration`,
  `function_declaration`, `method_definition`, `interface_declaration`, and `type_alias_declaration`.
- **Fallback**: Top-level statements that do not belong to a structural node (e.g., global variable declarations,
  standard imports) will be grouped into a `module_scope` chunk.
- **Output Schema**: `{ filePath: string, startLine: number, endLine: number, nodeType: string, content: string }`.

## 2. Core Memory Injection

Memories can have multiple string tags (e.g., `['preference', 'core']`).

- The `before_agent_start` hook in `src/shared/workflow.js` (or the mnemosyne extension) will intercept agent
  initialization.
- It will execute a deterministic query against both the global and local LanceDB memory tables for any record where
  `tags` includes `"core"`.
- The results will be silently prepended to the agent's system prompt to establish the foundational project DNA.

## 3. Tool Contracts & Examples

### `codebase_search`

- **Request**: `{ "query": "how are user passwords hashed?" }`
- **Response**:
  ```json
  [
      {
          "file": "src/utils/crypto.js",
          "lines": "14-28",
          "content": "export async function hashPassword(plainText) {..."
      }
  ]
  ```

### `memory_recall`

- **Request**: `{ "query": "database preferences", "scope": "global" }` (Note: scope can be `local`, `global`, or
  `both`).
- **Response**:
  ```json
  [
      {
          "id": "mem_9f8a7b",
          "content": "Prefers standard SQL over ORMs.",
          "tags": ["preference", "core"],
          "scope": "global",
          "score": 0.94
      },
      {
          "id": "mem_2c3d4e",
          "content": "Use LanceDB for local vector storage.",
          "tags": ["adr"],
          "scope": "local",
          "score": 0.88
      }
  ]
  ```

### `memory_add`

- **Request**:
  `{ "content": "Always use 20px grid for terminal UI designs.", "tags": ["design", "core"], "scope": "global" }`
- **Response**: `{ "status": "success", "id": "mem_5a6b7c" }`

### `memory_delete`

- **Request**: `{ "id": "mem_2c3d4e", "scope": "local" }`
- **Response**: `{ "status": "success", "message": "Memory mem_2c3d4e deleted." }`

### `memory_dump`

Used by the `sleep` command to allow the model to review and consolidate drifting context.

- **Request**: `{ "scope": "local" }`
- **Response**:
  ```json
  [
      { "id": "mem_111", "content": "...", "tags": ["core"] },
      { "id": "mem_222", "content": "...", "tags": ["bug-fix"] }
  ]
  ```
